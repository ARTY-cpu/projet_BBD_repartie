01 — Le contexte

On gère quoi exactement ?

Une entreprise de santé avec des cliniques et pharmacies dans deux villes du Maroc

Casablanca

Site S1

  • 1 clinique avec 3 médecins
  • 1 pharmacie qui vend les médicaments
  • 6 patients inscrits à Casablanca

Rabat

Site S2

  • 1 clinique avec 3 médecins
  • 1 pharmacie partenaire
  • 6 patients inscrits à Rabat

La question : Comment organiser ces données pour que chaque ville travaille vite en local, tout en gardant une vue d'ensemble ?

02 — Justification

Pourquoi ne pas tout mettre au même endroit ?

On compare les deux approches

Base centralisée
  • ✗ Si le serveur tombe, tout s'arrête
  • ✗ Rabat doit passer par le réseau à chaque requête → lent
  • ✗ Un seul serveur pour tout le monde → surchargé
Base répartie (notre choix)
  • ✓ Chaque ville accède à ses données directement
  • ✓ Si un site tombe, l'autre continue de fonctionner
  • ✓ La charge est répartie sur 2 serveurs

En résumé : les données de Casa restent à Casa, celles de Rabat restent à Rabat. On ne passe par le réseau que quand on a besoin d'une vue globale.

03 — Démarche

Comment on a procédé

On a suivi une approche classique en 3 étapes : du global vers le local

1

On conçoit le schéma global

On dessine toutes les tables comme si c'était une seule base → 9 tables

2

On découpe (fragmentation)

On sépare les données par ville (horizontal) et par usage (vertical)

3

On place les morceaux (allocation)

On décide quel serveur stocke quoi, et on connecte les deux avec FDW

PostgreSQL 16 postgres_fdw Docker Compose

04 — Fragmentation horizontale

On sépare les lignes par ville

Concrètement : les patients de Casa restent sur le serveur de Casa

Le principe est simple :

Patient_Casa = les patients où Ville = 'Casablanca'
Patient_Rabat = les patients où Ville = 'Rabat'

On fait pareil pour 6 tables :

Table On découpe sur… S1 Casa S2 Rabat
Patient Ville du patient ✓ ✓
Medecin Ville du médecin ✓ ✓
Stock Ville de la pharmacie ✓ ✓
Vente Ville de la pharmacie ✓ ✓
Consultation Via le médecin ✓ ✓
Prescription Via la consultation ✓ ✓
✓ Aucune donnée perdue (complétude)
✓ Pas de doublons (disjonction)
✓ On reconstruit avec UNION ALL

04 — Fragmentation verticale

On sépare les colonnes par usage

Pour la table Medicament, le médecin et le pharmacien n'ont pas besoin des mêmes infos

Medicament_Base → S1 (Casa)

Ce que le médecin a besoin de savoir

IdMedicament PK
NomMedicament
Forme (comprimé, sirop…)
Dosage (500mg, 1g…)
Medicament_Commercial → S2 (Rabat)

Ce que le pharmacien a besoin de savoir

IdMedicament PK
PrixUnitaire
Fabricant
VilleProduction

Pour reconstruire la table complète : on fait un JOIN entre les deux fragments sur l'IdMedicament

Attention : cette jointure passe par le réseau puisque les 2 morceaux sont sur des serveurs différents

05 — Transparence

L'utilisateur ne voit rien

Grâce aux vues et triggers, il manipule les données comme si tout était dans une seule base

Pour voir tous les patients (des 2 villes) : 
CREATE VIEW v_Patient AS
  SELECT * FROM Patient          -- données locales
  UNION ALL
  SELECT * FROM Patient_Rabat;   -- données de l'autre site (via FDW)
Pour insérer un patient — le routage est automatique : 
-- L'utilisateur écrit juste :
INSERT INTO v_Patient VALUES (13, 'Dupont', 'Marie', ..., 'Casablanca', ...);

-- Le trigger regarde la ville et envoie automatiquement :
-- → 'Casablanca' → table locale Patient
-- → 'Rabat' → table distante Patient_Rabat (via FDW)
Pour voir le catalogue médicament complet : 
CREATE VIEW v_Medicament AS
  SELECT b.*, c.PrixUnitaire, c.Fabricant, c.VilleProd
  FROM Medicament_Base b
  JOIN Medicament_Commercial c ON b.IdMedicament = c.IdMedicament;

07 — Bilan

Ce qui marche bien, et ce qu'on peut améliorer

Un regard honnête sur notre architecture

Ce qui marche bien
Transparent pour l'utilisateur — il fait un SELECT sur une vue, il ne sait pas que c'est réparti
Rapide en local — un médecin de Casa ne passe jamais par le réseau pour ses patients
Autonome — chaque site fonctionne seul pour ses opérations courantes
Extensible — on peut ajouter un site Marrakech (S3) plus tard
Ce qu'on peut améliorer
Pas de copie de secours — si Rabat tombe, ses données sont inaccessibles
Pas de clé étrangère cross-site — la cohérence repose sur l'application
Jointures distantes = lent — si le réseau est lent, les vues globales aussi
IDs manuels — on utilise des plages (1-100 / 101-200), fragile à long terme

07 — Scénarios

Et si… ?

Deux questions qu'on pourrait nous poser

Si le serveur de Rabat tombe ?
Casablanca continue à fonctionner pour ses données locales
Les vues globales ne marchent plus (erreur sur les tables distantes)
La vue médicament est bloquée (le fragment commercial est sur S2)

Solution : ajouter de la réplication pour avoir une copie de secours

Si on ajoute Marrakech (S3) ?
1. On crée une nouvelle base site_marrakech
2. On ajoute les connexions FDW (2 → 6 liens)
3. On met à jour les vues (UNION ALL avec 3 sources)
4. On ajoute 'Marrakech' dans les triggers de routage

C'est faisable mais au-delà de 3 sites, il vaudrait mieux utiliser Citus

Conclusion

Les 4 choses à retenir

1

Méthode claire — on part du schéma global, on découpe, puis on place. C'est la démarche descendante.

2

Deux types de découpe — horizontale (par ville, naturel) et verticale (par usage, plus rare mais pertinent ici).

3

Transparent pour l'utilisateur — les vues et triggers font tout le travail, il n'a pas besoin de savoir où sont les données.

4

Des compromis — répartir, c'est gagner en performance locale mais accepter plus de complexité (réseau, cohérence, pannes).
